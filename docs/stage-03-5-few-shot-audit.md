## 核心问题

现在的系统有一个**结构性矛盾**：

```
想做的事：LLM 泛化理解自然语言 → 生成 Cypher
实际做的事：在每个模块的 prompt 里硬塞领域知识来补偿 LLM 的不足
```

具体来说，当前管线的 4 个 LLM 调用（意图→规划→Cypher→回答）都是**单次调用 + 单次输出**。LLM 没有机会：

1. **发现领域**：不能主动查"图里有哪些项目类型？"只能接受 prompt 里预装的枚举
2. **验证假设**：不能试跑一条 Cypher 看看结果再修正，只能盲生成
3. **逐步推理**：不能先查"万科有哪些项目"再查"这些项目用了什么设备"，只能一次性生成完整 Cypher
4. **纠正错误**：虽然有重试，但重试只是"再生成一次"，不知道上次错在哪

为了弥补这些能力缺失，我们不得不在每个 prompt 里加入越来越多的领域知识、消歧规则、特殊 case 处理。**这就是为什么 prompt 会领域绑定 — 不是写法问题，是架构问题。**

## 结论

这强化了之前的判断：

| 路线 | 投入 | 解决的问题 | 留下的问题 |
|---|---|---|---|
| 继续修补当前管线 | 中 | 提升到 ~90% 通过率 | 领域绑定、few-shot 作弊、不可泛化、无多轮能力 |
| **Agent 架构** | 大 | 领域发现、自纠错、多轮对话、可泛化 | 需要验证 Qwen tool calling 可靠性 |

在 Agent 架构下，当前审计出来的每个绑定点都有自然解法：

| 当前绑定 | Agent 解法 |
|---|---|
| router.py `filter_keys` 硬编码 | Agent 不需要独立 router，工具选择即意图 |
| `"不要误归类为 project_name"` 消歧 | Agent 调用 `list_domain_values("project_names")` 自行判断 |
| `"替代" in question` 特殊分支 | Agent 读取 schema 发现 CAN_REPLACE 关系，自行决定查询方向 |
| planner.py `"最低设备+替代方案"` 规则 | Agent 的 ReAct 循环天然支持：执行→观察→下一步 |
| few_shots.yaml 97% 重叠 | Agent 从 schema 模式生成查询，不依赖 few-shot 模板匹配 |

**建议行动**：跳过 Stage 3.4 的修补，直接启动 Agent 架构。当前 85% 的通过率虽然含水分，但验证了"LLM + Schema + Neo4j"这条路是可行的。接下来应该用正确的架构来兑现这个可行性，而不是在错误的架构上堆更多补丁。
